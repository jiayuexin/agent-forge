## 1.1 IAgent 统一接口

整个框架的核心契约，所有 Agent 都必须实现此接口。

AgentForge 中存在两种 Agent 形态：

- **ClientAgent**：运行在用户机器上的客户端应用，有状态，可被 Capability Hub 远程控制。
- **StatelessAgent**：由编排器在进程内实例化的无状态 Agent，所有行为由 LLM 根据能力清单编排。

两种形态共享 `IAgent` 接口，但各自扩展了专属能力（见 §1.2）。两种形态的分野、生命周期、部署方式与适用场景详见 [08-客户端Agent与无状态Agent.md](./08-客户端Agent与无状态Agent.md)。

```typescript
interface IAgent<TConfig extends AgentConfig = AgentConfig> {
  readonly id: string;
  readonly name: string;
  readonly role: string;
  readonly version: string;
  readonly capabilities: AgentCapability[];
  readonly status: AgentStatus;

  init(config?: TConfig): Promise<void>;
  execute(task: AgentTask): Promise<AgentResult>;
  stream(task: AgentTask): AsyncIterable<AgentStreamChunk>;
  destroy(): Promise<void>;
  use(plugin: IPlugin): this;
  on(event: AgentEvent, handler: EventHandler): this;
  off(event: AgentEvent, handler: EventHandler): this;
}

/** Agent 事件类型 — 字符串字面量联合,便于 IDE 自动补全 */
type AgentEvent =
  | 'agent:init'         // 初始化完成
  | 'agent:ready'        // 就绪,等待任务
  | 'agent:execute:start' // 开始执行任务
  | 'agent:execute:end'   // 任务执行完成
  | 'agent:tool:call'     // 工具被调用前
  | 'agent:tool:result'   // 工具返回结果
  | 'agent:llm:chunk'     // LLM 流式输出一个 chunk
  | 'agent:llm:error'     // LLM 调用失败
  | 'agent:error'         // Agent 内部错误
  | 'agent:destroy'      // 销毁前
  // ClientAgent 专属事件 —— 仅守护进程形态触发
  | 'agent:capability:installed' // 能力安装完成
  | 'agent:hub:connected'        // 已连接到 Capability Hub
  | 'agent:hub:disconnected';    // 与 Capability Hub 断开连接

/** 事件处理器签名 */
type EventHandler<T = unknown> = (payload: T) => void | Promise<void>;

/** Agent 流式输出 chunk — Agent 层聚合,IProvider 层 chunk 不会直接暴露 */
interface AgentStreamChunk {
  type: 'text' | 'tool_call' | 'tool_result' | 'thinking' | 'done' | 'error';
  content?: string;
  toolCall?: {
    name: string;
    args: Record<string, unknown>;
  };
  toolResult?: {
    name: string;
    output: unknown;
  };
  error?: { code: string; message: string };
  /** chunk 序号,从 0 开始;done 类型时不递增 */
  index: number;
  /** 当前累计 token 数(输入+输出) */
  tokensUsed?: { input: number; output: number };
}

enum AgentStatus {
  UNINITIALIZED = 'uninitialized',
  INITIALIZING = 'initializing',
  READY = 'ready',
  DAEMON_RUNNING = 'daemon-running', // ClientAgent 守护进程已启动,等待远程/本地任务
  RUNNING = 'running',
  ERROR = 'error',
  DESTROYED = 'destroyed',
}

/**
 * Agent 自身声明的能力 — Capability 的轻量形态。
 * 当 Agent 注册到 CapabilityRegistry 时，BaseAgent 会将其转换为完整的 Capability，转换规则如下：
 * - id: `${agentId}:${name}`（agentId 取自 IAgent.id，name 取自 AgentCapability.name）
 * - type: 'agent'
 * - name: AgentCapability.name
 * - description / inputSchema / outputSchema / riskLevel / sensitiveOperations: 1:1 透传
 */
interface AgentCapability {
  /** 能力唯一标识，kebab-case */
  name: string;
  /** 能力描述，用于 PlannerAgent 理解和 Capability Hub 展示 */
  description: string;
  /** 输入参数 JSON Schema */
  inputSchema?: JSONSchema;
  /** 输出结果 JSON Schema */
  outputSchema?: JSONSchema;
  /** 能力风险等级 */
  riskLevel?: 'low' | 'medium' | 'high';
  /** 是否涉及敏感操作（如资金、权限变更、数据删除） */
  sensitiveOperations?: string[];
}

/** Agent 身份信息 —— IAgent 的只读属性 id / name / role / version 的来源 */
interface AgentIdentity {
  /** Agent 唯一标识；未提供时由 BaseAgent 自动生成 UUID */
  id?: string;
  /** Agent 名称（URL-safe，kebab-case） */
  name: string;
  /** Agent 岗位角色描述 */
  role: string;
  /** 语义化版本 */
  version: string;
}
```

### BaseAgent 抽象类

`BaseAgent` 是实现 `IAgent` 的抽象基类，内置生命周期、事件总线、插件加载与中间件链。具体 Agent 只需实现 `doInit`（可选）、`doExecute` 与可选的 `doStream`。

**初始化契约**：
- `constructor(config?)` 仅保存配置，不触发实际初始化；允许子类在构造时传入默认配置。
- `init(config)` 是正式初始化入口，完成状态迁移、Provider 创建、插件与中间件加载，最后调用 `doInit`。
- `AgentFramework.run()` / `orchestrate()` 在首次使用时会自动对未初始化的 Agent 调用 `init()`。

```typescript
abstract class BaseAgent<TConfig extends AgentConfig = AgentConfig> implements IAgent<TConfig> {
  constructor(config?: TConfig);

  readonly id: string;
  readonly name: string;
  readonly role: string;
  readonly version: string;
  readonly capabilities: AgentCapability[];
  readonly status: AgentStatus;

  /**
   * 初始化入口；完成状态迁移、Provider 创建、插件与中间件加载，最后调用 doInit。
   * 若 constructor 已传入 config，此处可省略 config 参数，BaseAgent 会复用构造时的配置。
   */
  async init(config?: TConfig): Promise<void>;

  /** 子类可覆盖的初始化钩子；默认空实现 */
  protected async doInit?(): Promise<void>;

  /** 执行入口；负责中间件执行、LLM 调用、工具调度，最后调用 doExecute */
  async execute(task: AgentTask): Promise<AgentResult>;

  /** 子类必须实现的核心执行逻辑 */
  protected abstract doExecute(task: AgentTask): Promise<AgentResult>;

  /** 流式执行入口 */
  async stream(task: AgentTask): AsyncIterable<AgentStreamChunk>;

  /** 子类可选实现的流式执行逻辑；默认将 doExecute 结果包装为单个 chunk */
  protected async doStream?(task: AgentTask): AsyncIterable<AgentStreamChunk>;

  /** 销毁 Agent，释放资源 */
  async destroy(): Promise<void>;

  use(plugin: IPlugin): this;
  on(event: AgentEvent, handler: EventHandler): this;
  off(event: AgentEvent, handler: EventHandler): this;
}
```

## 1.4 AgentConfig 配置模型

`AgentConfig` 是 ClientAgent 与 StatelessAgent 的基础配置模型。`identity` 字段提供 `IAgent` 所需的 `id / name / role / version`；未提供 `id` 时由 `BaseAgent` 自动生成。具体形态的配置（`ClientAgentConfig`、`StatelessAgentConfig`）继承自本接口（详见 §1.2）。

支持多 Provider 联合类型，根据 `provider` 字段区分不同配置。

```typescript
type ProviderType = 'openai' | 'anthropic' | 'ollama';

interface BaseModelParams {
  temperature?: number;
  maxTokens?: number;
  timeout?: number;
}

interface OpenAIModelConfig extends BaseModelParams {
  provider: Extract<ProviderType, 'openai'>;
  modelName: string;
  apiKey: string;
  baseUrl?: string;
  organization?: string;
}

interface AnthropicModelConfig extends BaseModelParams {
  provider: Extract<ProviderType, 'anthropic'>;
  modelName: string;
  apiKey: string;
  baseUrl?: string;
  anthropicVersion?: string;
}

interface OllamaModelConfig extends BaseModelParams {
  provider: Extract<ProviderType, 'ollama'>;
  modelName: string;
  baseUrl?: string;
}

interface CustomModelConfig extends BaseModelParams {
  provider: string;
  modelName: string;
  apiKey?: string;
  baseUrl?: string;
  [key: string]: any;
}

type ModelConfig =
  | OpenAIModelConfig
  | AnthropicModelConfig
  | OllamaModelConfig
  | CustomModelConfig;

interface AgentConfig {
  /** Agent 身份信息 —— IAgent 只读属性 id / name / role / version 的来源 */
  identity: AgentIdentity;
  model: ModelConfig;
  systemPrompt: string;
  /** Agent 能力清单；未提供时 BaseAgent 默认使用空数组 */
  capabilities?: AgentCapability[];
  tools?: ToolDefinition[];
  middlewares?: MiddlewareConfig[];
  custom?: Record<string, any>;
}

/** 工具定义 — Agent 可调用的工具,映射到各 Provider 的 Function Calling 格式 */
interface ToolDefinition {
  /** 工具名,使用 kebab-case(如 query-order, create-refund) */
  name: string;
  /** 工具描述(供 LLM 理解何时调用) */
  description: string;
  /** 参数 JSON Schema */
  parameters: JSONSchema;
  /** 是否为必须工具(无论 LLM 是否主动调用都注入到 system prompt) */
  required?: boolean;
  /**
   * 工具执行函数；Provider function calling 后由 Agent 调用。
   * 校验规则：handler 与 endpointType/endpoint 至少存在一组；同时存在时 handler 优先。
   */
  handler?: ToolHandler;
  /**
   * 工具执行端点类型。
   * 当 handler 缺失时，运行时通过 endpointType + endpoint 解析执行目标。
   * ToolCapability → ToolDefinition 的映射规则见 09-能力市场与下发.md §9.2。
   */
  endpointType?: 'local-command' | 'local-function' | 'http' | 'remote-agent';
  /** 端点声明；与 endpointType 配合使用 */
  endpoint?: {
    /** 本地函数引用、命令模板或 HTTP 路径 */
    target: string;
    /** 调用方式 */
    method?: 'exec' | 'call' | 'post' | 'get';
  };
}

/** 工具执行函数签名 */
type ToolHandler = (
  args: Record<string, unknown>,
  context: ToolContext
) => Promise<unknown>;

/** 工具执行上下文 */
interface ToolContext {
  /** 当前 Agent 实例 */
  agent: IAgent;
  /** 日志实例 */
  logger: Logger;
}

/** 中间件配置 */
interface MiddlewareConfig {
  /** 中间件名 */
  name: string;
  /** 启用/禁用 */
  enabled?: boolean;
  /** 中间件选项 */
  options?: Record<string, unknown>;
}
```

## 1.6 模型注册表（ModelRegistry）

> 编排器核心能力：集中管理多个模型端点，Agent 只需引用模型名即可。

### 设计思路

实际业务中，一个项目可能同时使用多个模型服务：

```
OpenAI 官方 API    → gpt-4o, gpt-4o-mini, o1
私有代理           → gpt-4o（通过代理访问）
Azure OpenAI      → gpt-4o（企业版）
本地 Ollama       → qwen2.5:14b, deepseek-r1:14b
DeepSeek API      → deepseek-chat, deepseek-reasoner
```

同一个模型名（如 `gpt-4o`）可能存在于多个端点。**模型注册表**让编排器统一管理这些端点，Agent 只需要写模型名，编排器自动路由。

### 类型定义

```typescript
/** 单个模型端点 — 一个 baseUrl 下可注册多个模型 */
interface ModelEndpoint {
  /** 端点标识符，用于 agent 引用 */
  id: string;
  /** 端点地址 */
  baseUrl: string;
  /** Provider 类型 */
  provider: 'openai' | 'anthropic' | 'ollama' | string;
  /** API Key（openai/anthropic 需要） */
  apiKey?: string;
  /** 额外参数（anthropicVersion, organization 等） */
  extra?: Record<string, any>;
  /** 该端点下注册的模型列表 */
  models: string[];
}

/** 模型注册表 — 框架初始化时一次性配置 */
interface ModelRegistry {
  /** 所有端点 */
  endpoints: ModelEndpoint[];
  /** 默认端点 id（Agent 不指定模型时的回退） */
  defaultEndpoint?: string;
  /** 默认模型名 */
  defaultModel?: string;
}

/** Agent 引用模型的方式 — 只需模型名 + 可选端点 id */
interface ModelRef {
  /** 模型名（如 'gpt-4o', 'qwen2.5:14b'） */
  model: string;
  /** 指定使用哪个端点（可选，不指定时自动匹配第一个注册了该模型的端点） */
  endpoint?: string;
}
```

### 使用方式

**框架初始化 — 注册所有模型端点：**

```typescript
const framework = new AgentFramework({
  modelRegistry: {
    endpoints: [
      {
        id: 'openai-official',
        baseUrl: 'https://api.openai.com/v1',
        provider: 'openai',
        apiKey: process.env.OPENAI_API_KEY,
        models: ['gpt-4o', 'gpt-4o-mini', 'o1'],
      },
      {
        id: 'openai-proxy',
        baseUrl: 'https://my-proxy.example.com/v1',
        provider: 'openai',
        apiKey: process.env.PROXY_API_KEY,
        models: ['gpt-4o'],
      },
      {
        id: 'ollama-local',
        baseUrl: 'http://localhost:11434',
        provider: 'ollama',
        models: ['qwen2.5:14b', 'deepseek-r1:14b'],
      },
      {
        id: 'deepseek-api',
        baseUrl: 'https://api.deepseek.com/v1',
        provider: 'openai',   // DeepSeek 兼容 OpenAI 协议
        apiKey: process.env.DEEPSEEK_API_KEY,
        models: ['deepseek-chat', 'deepseek-reasoner'],
      },
    ],
    defaultEndpoint: 'openai-official',
    defaultModel: 'gpt-4o',
  },
});
```

**Pipeline 编排 — Agent 只写模型名：**

```typescript
await framework
  .pipeline('智能客服')
  .config({
    defaultModel: 'gpt-4o',          // 流水线级默认
  })
```

### 模型解析规则

编排器解析 Agent 的模型引用时，按以下规则：

```
1. Agent 指定了 endpoint → 直接用该端点 + 模型名
2. Agent 只指定了 model 名 → 在注册表中查找第一个注册了该模型的端点
3. Agent 都没指定 → 用 Pipeline 级 defaultModel，路由到 defaultEndpoint
4. 找不到匹配的端点 → 报错：ModelNotFoundError
```

### Agent 单独运行

Agent 不在 Pipeline 中单独跑时，可以简单配置一个端点：

```typescript
    provider: 'ollama',
    modelName: 'qwen2.5:14b',
    baseUrl: 'http://localhost:11434',

```

```typescript
interface PipelineControlSignal {
```

### Provider Tool Call 格式适配

各 Provider 的 Function Calling 格式不同,IProvider 实现需负责适配:

| Provider | 工具定义格式 | 工具调用格式 | 适配层位置 |
|---|---|---|---|
| OpenAI | `tools[].function` (name/description/parameters) | `tool_calls[].function` (name/arguments) | `OpenAIProvider.ts` 内部转换 |
| Anthropic | `tools[]` (name/description/input_schema) | `content[].type=tool_use` (name/input) | `AnthropicProvider.ts` 内部转换 |
| Ollama | 兼容 OpenAI 格式 | 兼容 OpenAI 格式 | 直接透传 |

框架内部统一使用 `ToolDefinition`(kebab-case name + JSONSchema parameters)和 `ToolCallRequest`(name + args + callId),各 Provider 适配层负责 bidirectional mapping。

## 1.14 模型驱动编排类型

> 编排框架内部包含一个规划 Agent（PlannerAgent），它基于系统已注册的能力（Capability）动态规划工作流，再由 PlanExecutor 执行。
>
> 设计详情参见 [04-集成与编排.md](./04-集成与编排.md) 和 [08-客户端Agent与无状态Agent.md](./08-客户端Agent与无状态Agent.md)。

### Capability（能力）

```typescript
/** 能力类型 — 编排器可调用的最小单元 */
type CapabilityType = 'agent' | 'tool' | 'skill' | 'plugin' | 'remote-agent';

/** remote-agent 表示一个可通过网络调用的 ClientAgent 节点 */
interface RemoteAgentCapability extends Capability {
  type: 'remote-agent';
  /** 节点 ID */
  nodeId: string;
  /** 节点地址或 Capability Hub 路由标识 */
  endpoint?: string;
}

/**
 * 能力分层模型（Tool / Skill / Plugin）的定义格式、版本管理、下发协议与本地缓存规则
 * 详见 09-能力市场与下发.md。
 */

/** 能力元数据 — 描述一个可被编排器调用的能力 */
interface Capability {
  /** 唯一标识，全局唯一 */
  id: string;
  /** 能力类型 */
  type: CapabilityType;
  /** 展示名称 */
  name: string;
  /** 自然语言描述：做什么、何时调用、输入输出是什么 */
  description: string;
  /** 输入参数 JSON Schema */
  inputSchema?: JSONSchema;
  /** 输出结果 JSON Schema */
  outputSchema?: JSONSchema;
  /** 标签，用于筛选和分类 */
  tags?: string[];
  /** 版本 */
  version?: string;
  /** 依赖的其他能力 id */
  dependencies?: string[];
  /** 风险等级 — 用于判断是否触发人工确认 */
  riskLevel?: 'low' | 'medium' | 'high';
  /** 敏感操作标签 — 如 refund, contract-sign, payment */
  sensitiveOperations?: string[];
}

/** 能力注册表选项 */
interface CapabilityRegistryOptions {
  /** 能力 ID 冲突时的处理策略 */
  onConflict?: 'overwrite' | 'ignore' | 'throw';
}

/** 能力注册表 — 发现、描述和管理系统能力 */
interface CapabilityRegistry {
  /** 注册一个能力；options 用于指定冲突处理策略，未指定时使用注册表全局默认策略 */
  register(capability: Capability, options?: CapabilityRegistryOptions): void;
  /** 注销一个能力 */
  unregister(id: string): void;
  /** 列出能力，支持按类型和标签筛选 */
  list(filters?: { type?: CapabilityType | CapabilityType[]; tags?: string[] }): Capability[];
  /** 获取单个能力 */
  get(id: string): Capability | undefined;
  /** 生成给 PlannerAgent 的能力清单文本 */
  toPrompt(): string;
}
```
