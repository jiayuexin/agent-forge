## 4.1 集成模式选择

新方向下，AgentForge 提供三种使用形态：

| 形态 | 适用场景 | 用户 | 关键组件 |
|---|---|---|---|
| **ClientAgent 客户端运行** | 需要长期运行在用户机器、调用本地资源、可远程扩展 | 终端用户、团队管理员 | `@agentforge/cli`、`@agentforge/runtime-client` |
| **SDK 编排 StatelessAgent** | 需要编排多个无状态 Agent 完成复杂任务 | 开发者、系统架构师 | `@agentforge/sdk` |
| **自定义 Capability Hub** | 需要自建能力中心和管控面 | 企业 IT 团队 | `@agentforge/dashboard`、`@agentforge/sdk` |

> 旧方向中的 "npm 包集成" 和 "HTTP 服务集成" 不再作为主要生产路径。ClientAgent 内部可能暴露本地 HTTP 接口用于调试，但不作为独立部署单元。

---

## 4.2 形态 1：ClientAgent 客户端运行

用户通过 CLI 生成并启动 ClientAgent，Agent 以后台守护进程形式运行在本机。

```bash
agentforge create "一个能执行 Git 命令的本地编程助手"
agentforge run ./client-agents/my-agent \
  --connect wss://hub.example.com \
  --token $AGENTFORGE_NODE_TOKEN
```

ClientAgent 启动后：

1. 初始化 `IClientAgent` 实例。
2. 加载本地能力缓存 `.agentforge/capabilities/`。
3. 连接 Capability Hub，注册为 Agent Node。
4. 启动心跳，等待远程任务或本地交互。

用户可以通过以下方式与 ClientAgent 交互：

- Capability Hub 的 Web 界面远程对话。
- ClientAgent 本地暴露的调试 HTTP 接口。
- 系统托盘/CLI 命令行。

---

## 4.3 形态 2：SDK 编排 StatelessAgent

开发者通过 `@agentforge/sdk` 编排多个 StatelessAgent，构建复杂工作流。

```typescript
import { AgentFramework } from '@agentforge/sdk';
import { GitAgent, ReviewerAgent } from './agents';

const framework = new AgentFramework();
framework.register('git', GitAgent);
framework.register('reviewer', ReviewerAgent);

// 模型驱动编排
const result = await framework.orchestrate({
  type: 'chat',
  input: { message: 'review 这段代码并生成 commit message' },
});
```

编排器内部：

1. `CapabilityRegistry` 发现已注册能力。
2. `PlannerAgent` 根据任务和能力清单生成 `ExecutionPlan`。
3. `PlanExecutor` 解析依赖、调度执行。
4. `Pipeline` 作为底层执行引擎运行每个步骤。

---

## 4.4 形态 3：自定义 Capability Hub

企业可以基于 `@agentforge/dashboard` 和 `@agentforge/sdk` 构建自定义 Capability Hub：

- 管理内部 Tool / Skill / Plugin 定义。
- 控制哪些 ClientAgent 可以接收哪些能力。
- 审计能力下发记录和 Agent 执行日志。
- 集成内部认证系统。

```typescript
import { DashboardServer } from '@agentforge/dashboard';

const hub = new DashboardServer({
  port: 8080,
  auth: { type: 'sso', provider: 'internal' },
  capabilityStore: { type: 'filesystem', path: './capabilities' },
});

await hub.start();
```

## 4.5 ClientAgent 与 StatelessAgent 协作

两种 Agent 可以协同工作：

```text
┌─────────────────────────────────────────┐
│           Capability Hub                │
│  · 能力管理 / 能力市场 / 能力下发        │
│  · ClientAgent 节点注册表                │
└─────────────────────────────────────────┘
              ↑↓ WebSocket
┌─────────────────────────────────────────┐
│         ClientAgent（用户机器）          │
│  · 守护进程                              │
│  · 本地命令执行（授权后）                 │
│  · 能力缓存                              │
└─────────────────────────────────────────┘
              ↑↓ 本地 HTTP/WSS 或 Hub 路由
┌─────────────────────────────────────────┐
│  SDK 进程内的 StatelessAgent 编排        │
│  · PlannerAgent 生成计划                 │
│  · PlanExecutor 调度执行                 │
│  · 可调用 ClientAgent 的远程能力          │
└─────────────────────────────────────────┘
```

### 协作示例

用户请求："帮我 review 这段代码，如果有问题就生成修复补丁并提交。"

1. `AgentFramework.orchestrate()` 被调用。
2. `CapabilityRegistry` 发现能力：
   - 本地 StatelessAgent：`reviewer`
   - 远程 ClientAgent：`client-dev-machine`，具备 `git-commit` 能力
3. `PlannerAgent` 生成计划：
   - Step 1: 调用 `reviewer` 进行代码审查
   - Step 2: 调用 `client-dev-machine` 的 `git-commit` 能力提交补丁
4. `PlanExecutor` 执行计划。
5. ClientAgent 在执行 `git-commit` 前，根据 `security.json` 进行本地确认。

## 4.6 能力注册表（CapabilityRegistry）

`CapabilityRegistry` 是编排器发现、描述、管理系统可调用能力的注册表。

### 能力类型

```typescript
type CapabilityType = 'agent' | 'tool' | 'skill' | 'plugin' | 'remote-agent';
```

| 类型 | 说明 |
|---|---|
| `agent` | 本地注册的 StatelessAgent |
| `tool` | 单个工具，如 `query-order` |
| `skill` | 一组相关 Tool + Prompt 模板，如 `refund-skill` |
| `plugin` | 可扩展代码包，如日志插件、认证插件 |
| `remote-agent` | 可通过网络调用的 ClientAgent 节点 |

### 注册表接口

```typescript
interface CapabilityRegistry {
  register(capability: Capability): void;
  unregister(id: string): void;
  list(filters?: { type?: CapabilityType | CapabilityType[]; tags?: string[] }): Capability[];
  get(id: string): Capability | undefined;
  toPrompt(): string;
}
```

### 能力发现机制

1. **Agent 自动发现**：`AgentFramework.register()` 时读取 Agent 类的 `static capability`。
2. **Tool 自动发现**：通过 `PluginContext.registerTool()` 注册的工具自动进入注册表。
3. **Skill 显式注册**：`framework.discovery.register({ type: 'skill', ... })`。
4. **Plugin 批量注册**：插件可实现 `capabilities()` 方法一次性注册多个能力。
5. **远程 Agent 注册**：ClientAgent 连接 Capability Hub 后，Hub 将其能力注入注册表。
6. **冲突处理**：同 ID 不同版本时保留高版本，可配置 `onConflict: 'overwrite' | 'ignore' | 'throw'`。

## 4.7 模型驱动编排架构

```text
用户任务
    ↓
CapabilityRegistry（能力清单）
    ↓
PlannerAgent（生成 ExecutionPlan）
    ↓
PlanExecutor（解析依赖、调度执行）
    ↓
Pipeline（底层串行/并行执行引擎）
    ↓
StatelessAgent / ClientAgent / Tool
```

### PlannerAgent

`PlannerAgent` 本身是一个 StatelessAgent，负责根据任务描述和可用能力生成执行计划。

```typescript
class PlannerAgent extends BaseAgent<PlannerConfig> {
  protected async doExecute(task: AgentTask): Promise<AgentResult> {
    const capabilities = this.registry.findRelevant(task.input.message);
    const plan = await this.llm.chat(buildPlanningPrompt(task, capabilities));
    return {
      success: true,
      output: { structured: plan },
    };
  }
}
```

### ExecutionPlan

```typescript
interface ExecutionPlan {
  id: string;
  task: AgentTask;
  steps: PlanStep[];
  metadata: {
    estimatedCost?: number;
    requireApproval?: boolean;
  };
}

interface PlanStep {
  id: string;
  name: string;
  capabilityId: string;
  task: AgentTask;
  dependsOn?: string[];
  outputsTo?: string;
}
```

### PlanExecutor

`PlanExecutor` 负责：

1. 解析 `ExecutionPlan` 中的依赖关系。
2. 拓扑排序，决定哪些步骤可以并行。
3. 替换变量引用（上一步的输出给下一步用）。
4. 对 StatelessAgent 直接进程内调用。
5. 对 ClientAgent 通过 Capability Hub 路由下发任务。
6. 某一步失败后，决定是否触发重新规划。

## 4.8 Pipeline：底层执行引擎

`Pipeline` 是确定性的底层执行引擎，适合固定流程：

```typescript
const result = await framework
  .pipeline('智能客服')
  .add('service', { task: '分析用户意图' })
  .branch((prev) => {
    const { intent } = prev.output.structured;
    if (intent === 'refund') return { agent: 'sales', task: '处理退款' };
    return { agent: 'service', task: '解答问题' };
  })
  .run();
```

支持：

- 串行 `.add()`
- 并行 `.parallel()`
- 条件分支 `.branch()`
- 步骤回退 `.back()`
- 分叉合并 `.fork()`

## 4.9 编排模式对比与选择边界

| 模式 | 适用场景 | 可控性 | 成本 |
|---|---|---|---|
| **模型驱动编排** | 开放任务、动态流程、复杂决策 | 中 | 高（每步规划都调 LLM） |
| **Pipeline 流水线** | 固定流程、需审计、成本敏感 | 高 | 低 |
| **ClientAgent 远程控制** | 需要本地资源、长期运行、人工介入 | 高 | 中 |
| **事件总线** | 异步通知、一对多广播 | 低 | 低 |
| **直接调用** | 简单 A 调 B、一次性场景 | 高 | 低 |

### 默认选择规则

**使用模型驱动编排（`framework.orchestrate`）的条件：**

- 任务为开放描述，未显式指定执行步骤。
- 能力注册表中存在 ≥2 个与任务相关的已注册能力。
- 用户未在 `OrchestrateOptions` 中禁用动态规划。

**直接使用固定 `Pipeline`（`framework.pipeline`）的条件：**

- 流程固定，步骤之间的依赖关系已知。
- 执行过程需严格审计或成本敏感。
- 用户显式调用 `framework.pipeline()`。

**Fallback 到固定 Pipeline 的触发条件：**

- `PlannerAgent` 连续无法生成合法计划，达到 `maxReplanAttempts` 上限。
- 任务被显式标记为 `fixedWorkflow: true`。
- 规划出的所有步骤均属于同一已知 Pipeline。

## 4.10 SDK 核心 API

```typescript
interface AgentFramework {
  // Agent 注册与管理
  register(name: string, AgentClass: new () => IAgent, capability?: Partial<Capability>): this;
  get(name: string): IAgent;
  init(config?: FrameworkConfig): Promise<void>;
  run(name: string, task: AgentTask): Promise<AgentResult>;

  // 能力发现
  readonly discovery: CapabilityRegistry;

  // 模型驱动编排
  plan(task: AgentTask, options?: PlanOptions): Promise<ExecutionPlan>;
  executePlan(plan: ExecutionPlan): Promise<PlanResult>;
  orchestrate(task: AgentTask, options?: OrchestrateOptions): Promise<PlanResult>;

  // 底层 Pipeline
  pipeline(name?: string): Pipeline;

  // ClientAgent 连接
  connectToClientAgent(nodeId: string, hubUrl: string, token: string): Promise<IClientAgentProxy>;

  // 事件总线
  on(event: string, handler: Function): this;
  once(event: string, handler: Function): this;
  off(event: string, handler: Function): this;
  emit(event: string, data: any): void;
}
```

## 4.11 重新规划策略

当 PlanStep 执行失败时，PlanExecutor 捕获错误并触发重新规划：

```typescript
interface PlanContext {
  task: AgentTask;
  plan: ExecutionPlan;
  failedStep: StepResult;
  completedSteps: StepResult[];
  remainingSteps: PlanStep[];
  userInput: Record<string, unknown>;
  attemptNumber: number;
}
```

重新规划流程：

1. 将失败步骤、已完成步骤、错误信息输入 `PlannerAgent.replan()`。
2. `PlannerAgent` 生成新的 `ExecutionPlan`。
3. `PlanExecutor` 执行新计划。
4. 达到 `maxReplanAttempts` 上限后返回 `AgentResult.error`。

---

## 4.12 人工确认机制

当计划满足以下任一条件时，触发人工确认：

- `requireApproval: true`
- 计划包含 `high` 风险能力
- 计划包含 `sensitiveOperations`

确认流程：

1. `PlanExecutor` 调用 `ApprovalHandler(plan)`。
2. Capability Hub 或宿主应用展示计划并等待用户确认。
3. 用户批准后继续执行；用户拒绝则返回错误。

对于 ClientAgent 的敏感操作，即使计划已批准，ClientAgent 本地仍会进行二次确认。
