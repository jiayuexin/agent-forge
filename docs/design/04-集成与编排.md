# 5. Agent 编排协作

> 核心理念：> 1. 每个 Agent 是独立 npm 包，互相不认识。
> 2. **编排器本身也是一个 Agent**（OrchestratorAgent / PlannerAgent），它基于系统已注册的能力动态规划工作流。
> 3. `Pipeline` 从"主要编排接口"退化为"底层执行引擎"，负责确定性调度（串行/并行/依赖管理）。

## 5.1 模型驱动编排架构

```
┌─────────────────────────────────────────────────────────────────────┐
│                         宿主项目 (你的业务系统)                        │
│                                                                      │
│   import { AgentFramework } from '@agentforge/sdk';                 │
│                                                                      │
│   ┌────────────────────────────────────────────────────────────┐    │
│   │              @agentforge/sdk（编排中心）                     │    │
│   │  ┌─────────────────┐    ┌──────────────┐    ┌───────────┐  │    │
│   │  │ CapabilityRegistry │──→│ PlannerAgent │──→│ PlanExecutor│  │    │
│   │  └─────────────────┘    └──────────────┘    └─────┬─────┘  │    │
│   │                                                   │        │    │
│   │                                          ┌────────▼────────┐│    │
│   │                                          │   Pipeline      ││    │
│   │                                          │ （底层执行引擎）  ││    │
│   │                                          └────────┬────────┘│    │
│   └───────────────────────────────────────────────────┼─────────┘    │
│                                                       │              │
│   ┌──────────┐  ┌────────────┐  ┌────────────┐     ┌─┴──────────┐   │
│   │ 客服Agent │  │ 销售Agent   │  │ 数据Agent   │◄────┤ Tool/Skill │   │
│   │ (npm包)   │  │ (npm包)     │  │ (npm包)     │     │   Plugin   │   │
│   └──────────┘  └────────────┘  └────────────┘     └────────────┘   │
└─────────────────────────────────────────────────────────────────────┘
```

### 三层编排模型

| 层级 | 角色 | 关键类 | 职责 |
|---|---|---|---|
| 能力层 | CapabilityRegistry | `CapabilityRegistry` | 发现、描述系统里所有可用能力 |
| 决策层 | PlannerAgent | `PlannerAgent` / `OrchestratorAgent` | 基于能力和任务，生成执行计划 |
| 执行层 | PlanExecutor | `PlanExecutor` + `Pipeline` | 按计划调度 Agent/Tool/Skill 执行 |

## 5.2 能力注册表（CapabilityRegistry）

### 什么是能力

系统里所有可被编排器调用的单元都叫**能力（Capability）**：

- **Agent**：完整的 Agent 实例，如 `customer-service`、`sales-assistant`
- **Tool**：单个工具函数，如 `query-order`、`create-refund`
- **Skill**：预定义的能力组合，可以是一个 Agent 的常用模式，也可以是多个 Tool 的封装
- **Plugin**：通过插件机制注册的能力包，可能同时暴露 Tool 和 Middleware

### 能力元数据

每个能力在注册时必须提供足够的描述信息，供 PlannerAgent 理解何时调用：

```typescript
interface Capability {
  id: string;                    // 唯一标识
  type: 'agent' | 'tool' | 'skill' | 'plugin';
  name: string;                  // 展示名称
  description: string;           // 自然语言描述：做什么、何时用
  inputSchema?: JSONSchema;      // 输入参数 Schema
  outputSchema?: JSONSchema;     // 输出结果 Schema
  tags?: string[];               // 标签，如 ['customer-service', 'order']
  version?: string;              // 版本
  dependencies?: string[];       // 依赖的其他能力 id
}

interface CapabilityRegistry {
  register(capability: Capability): void;
  unregister(id: string): void;
  list(filters?: { type?: string; tags?: string[] }): Capability[];
  get(id: string): Capability | undefined;
  toPrompt(): string;            // 输出给 PlannerAgent 的能力清单文本
}
```

### 能力发现方式

```typescript
const framework = new AgentFramework();

// 1. 注册 Agent，自动发现其 capability
framework.register('service', CustomerServiceAgent, {
  description: '处理用户咨询、投诉、退换货等客服场景',
  tags: ['customer-service'],
});

// 2. 注册单个 Tool
framework.discovery.register({
  id: 'query-order',
  type: 'tool',
  name: '查询订单',
  description: '根据订单号查询订单状态和物流信息',
  inputSchema: { type: 'object', properties: { orderId: { type: 'string' } }, required: ['orderId'] },
});

// 3. 注册 Skill（能力组合）
framework.discovery.register({
  id: 'handle-refund',
  type: 'skill',
  name: '处理退款',
  description: '完整的退款处理流程：查询订单 → 验证条件 → 创建退款 → 通知用户',
  dependencies: ['query-order', 'create-refund', 'send-notification'],
});

// 4. 插件安装时自动注册能力
agent.use(new RefundPlugin()); // Plugin 内部通过 PluginContext 注册 tool/capability
```

### 给 PlannerAgent 的能力清单

`CapabilityRegistry.toPrompt()` 会生成类似这样的文本：

```
可用能力：

[agent] customer-service
  描述：处理用户咨询、投诉、退换货等客服场景
  输入：{ message: string, userId?: string }
  输出：{ content: string, structured?: object }

[tool] query-order
  描述：根据订单号查询订单状态和物流信息
  输入：{ orderId: string }
  输出：{ status: string, logistics?: object }

[skill] handle-refund
  描述：完整的退款处理流程
  依赖：query-order, create-refund, send-notification
```

## 5.3 规划 Agent（PlannerAgent / OrchestratorAgent）

### 职责

PlannerAgent 是一个特殊的 Agent，它不直接处理业务任务，而是**为业务任务生成执行计划**。

输入：用户任务 + 能力清单 + 可选约束
输出：结构化的 `ExecutionPlan`

### 执行计划类型

```typescript
interface ExecutionPlan {
  goal: string;                  // 任务目标
  capabilitiesUsed: string[];    // 计划中使用的能力 id 列表
  steps: PlanStep[];             // 执行步骤
  constraints?: {
    maxSteps?: number;           // 最大步骤数
    requireApproval?: boolean;   // 执行前是否需要人工确认
    timeout?: number;            // 总超时
  };
}

interface PlanStep {
  id: string;                    // 步骤唯一 id
  name: string;                  // 步骤名称（用于日志和展示）
  capability: string;            // 调用的能力 id
  type: 'agent' | 'tool' | 'skill';
  task: string;                  // 给该能力的自然语言指令
  input: Record<string, any>;   // 输入数据（可引用上游步骤输出）
  dependsOn?: string[];          // 依赖的步骤 id
  outputAs?: string;             // 输出变量名，供下游引用
  fallback?: string;             // 失败时回退到哪个能力（可选）
}

interface PlanResult {
  success: boolean;
  output: AgentResult['output'];
  plan: ExecutionPlan;
  stepResults: StepResult[];
  totalDuration: number;
}

interface StepResult {
  stepId: string;
  success: boolean;
  output: any;
  duration: number;
  error?: { code: string; message: string };
}
```

### PlannerAgent 的 System Prompt 设计原则

1. **列出所有可用能力**：包括能力类型、描述、输入输出
2. **要求逐步推理**：先分析任务，再决定调用顺序
3. **输出结构化 JSON**：必须可解析为 `ExecutionPlan`
4. **明确依赖关系**：哪些步骤必须等前面步骤完成
5. **失败回退**：为关键步骤指定 fallback 能力

示例 prompt 片段：

```
你是一个任务编排器。请根据用户任务和以下可用能力，制定执行计划。

可用能力：
{capabilities}

要求：
1. 把任务拆分为最少必要步骤
2. 每个步骤只能调用一个能力
3. 用 dependsOn 表达步骤依赖
4. 输入可以引用上游步骤输出，使用 ${stepId.output.field} 语法
5. 输出必须是合法的 JSON，格式为 { goal, capabilitiesUsed, steps }
```

### 示例：用户想退款

用户输入："我要退掉订单 ORD-123"

PlannerAgent 生成的计划：

```json
{
  "goal": "处理用户退款请求",
```

### 重新规划（Re-planning）

当某一步失败时，PlanExecutor 把错误信息反馈给 PlannerAgent，PlannerAgent 可以生成新计划：

```typescript
async replan(failedStep: StepResult, context: PlanContext): Promise<ExecutionPlan>;
```

例如 `create-refund` 失败返回"订单已发货，不能直接退款"，PlannerAgent 可以重新规划：

```json
{
  "goal": "订单已发货，引导用户退货退款",
  "steps": [
    { "capability": "create-return-order", "task": "创建退货单" },
    { "capability": "customer-service", "task": "告知用户退货流程" }
  ]
}
```

## 5.4 计划执行器（PlanExecutor）

### 职责

PlanExecutor 负责把 PlannerAgent 生成的 `ExecutionPlan` 真正运行起来：

1. 解析步骤依赖图（DAG）
2. 调度无依赖的步骤并行执行
3. 收集步骤输出并绑定到下游输入
4. 处理失败和重新规划
5. 记录执行轨迹

### 执行流程

```
ExecutionPlan
    ↓
构建 DAG（根据 dependsOn）
    ↓
找出所有入度为 0 的步骤 → 并行执行
    ↓
收集输出 → 替换下游 input 中的变量引用
    ↓
重复直到所有步骤完成 或 遇到不可恢复错误
    ↓
返回 PlanResult
```

### 变量绑定

PlanStep 的 input 可以引用上游输出：

```typescript
{
  input: {
    orderId: "${verify.output.structured.orderId}",
    message: "${userInput.message}"
  }
}
```

PlanExecutor 在执行前会替换这些占位符为真实值。

### 与 Pipeline 的关系

PlanExecutor 内部可以复用 `Pipeline` 作为执行引擎：

- 串行依赖链 → `.add()`
- 无依赖的并行步骤 → `.parallel()`
- 复杂的 DAG → 拆成多个 Pipeline 阶段，每个阶段内并行

```typescript
class PlanExecutor {
  async execute(plan: ExecutionPlan): Promise<PlanResult> {
    const dag = this.buildDAG(plan.steps);
    const pipeline = this.framework.pipeline();

    // 简化的串行执行示例（实际按 DAG 调度）
    for (const step of this.topologicalSort(dag)) {
      pipeline.add(step.capability, {
        task: step.task,
        input: this.resolveInput(step.input, context),
      });
    }

    return pipeline.run();
  }
}
```

## 5.5 Pipeline：从主编排接口到底层执行引擎

### 角色转变

在模型驱动编排中，`Pipeline` 不再要求开发者手写 `.add() / .branch() / .parallel()`。它变成：

- **PlanExecutor 的底层调度工具**
- **开发者可选的高级接口**：用于固定、审计严格的流程

### 保留的能力

| 能力 | 保留？ | 说明 |
|---|---|---|
| `.add()` 串行 | ✅ | 执行计划中的依赖链 |
| `.parallel()` 并行 | ✅ | 执行计划中无依赖的步骤 |
| `.config()` | ✅ | Pipeline 级默认模型、超时等 |
| `.branch()` 条件分支 | ⚠️ 弱化 | 动态分支交给 PlannerAgent |
| `.intercept()` 拦截器 | ⚠️ 弱化 | 异常处理交给 PlanExecutor + PlannerAgent |
| `.fork()` 分叉 | ⚠️ 弱化 | 可用并行替代 |
| `back/jump` 控制信号 | ⚠️ 弱化 | 复杂流程控制通过重新规划实现 |

### 何时直接使用 Pipeline

以下情况仍然适合手写 Pipeline：

- 流程固定且需要严格审计（如金融审批、合同审批）
- 对成本敏感，不想每次调 PlannerAgent
- 需要人类精确控制每一步

```typescript
// 固定流程：仍然手写 Pipeline
const result = await framework
  .pipeline('合同审批')
  .add('draft', { task: '起草合同' })
  .add('review', { task: '法务审查' })
  .add('sign', { task: '生成签署文件' })
  .run();
```

## 5.8 编排模式对比

| 模式 | 耦合度 | 适用场景 | 配置复杂度 | 可控性 | 成本 |
|---|---|---|---|---|---|
| **模型驱动编排** | 低 | 开放任务、动态流程、复杂决策 | 中（需描述能力） | 中 | 高（每步规划都调 LLM） |
| Pipeline 流水线 | 中 | 固定流程、需审计、成本敏感 | 低 | 高 | 低 |
| 事件总线 | 低 | 异步通知、一对多广播 | 低 | 低 | 低 |
| 直接调用 | 高 | 简单 A 调 B、一次性场景 | 最低 | 高 | 低 |

## 5.9 SDK 核心 API

```typescript
interface AgentFramework {
  // Agent 注册与管理
  register(name: string, AgentClass: new () => IAgent, capability?: Partial<Capability>): this;
  get(name: string): IAgent;
  init(config?: FrameworkConfig): Promise<void>;
  run(name: string, task: AgentTask): Promise<AgentResult>;

  // 能力发现
  readonly discovery: CapabilityRegistry;

  // 模型驱动编排
  plan(task: AgentTask, options?: PlanOptions): Promise<ExecutionPlan>;
  executePlan(plan: ExecutionPlan): Promise<PlanResult>;
  orchestrate(task: AgentTask, options?: OrchestrateOptions): Promise<PlanResult>; // plan + execute

  // 底层 Pipeline（可选直接使用）
  pipeline(name?: string): Pipeline;

  // 事件总线
  on(event: string, handler: Function): this;
  once(event: string, handler: Function): this;
  off(event: string, handler: Function): this;
  emit(event: string, data: any): void;
}

interface OrchestrateOptions {
  requireApproval?: boolean;     // 执行前是否要求确认计划
  maxReplanAttempts?: number;    // 最大重新规划次数
  constraints?: PlanConstraints;
}

interface Pipeline {
  add(agentName: string, step: PipelineStep): this;
  parallel(steps: PipelineStep[]): this;
  config(options: PipelineConfig): this;
  run(): Promise<PipelineResult>;
}
```

```typescript
framework.register('service', CustomerServiceAgent);
framework.register('sales', SalesAssistantAgent);

await framework.init();

app.post('/api/chat', async (req, res) => {
  const result = await framework.orchestrate({
    type: 'chat',
    input: { message: req.body.message, userId: req.body.userId },
  });

  res.json({ reply: result.output.content });
});
```

PlannerAgent 会自动决定：
- 如果只是普通咨询 → 调用 `service`
- 如果有销售机会 → 先 `service` 分析意图，再 `sales` 推荐
- 如果涉及退款 → 调用 `service` + 相关 tools/skills

## 5.11 能力注册与发现机制

### 5.11.1 能力 ID 规则

- 全局唯一，使用 kebab-case
- 推荐前缀：`agent:`、`tool:`、`skill:`、`plugin:`
- 示例：`agent:customer-service`、`tool:query-order`、`skill:handle-refund`

### 5.11.2 Agent 自动发现

`AgentFramework.register()` 注册 Agent 时，自动向 `CapabilityRegistry` 注册对应能力。

```typescript
// 方式一：Agent 类静态属性
class CustomerServiceAgent extends BaseAgent {
  static capability: Capability = {
    id: 'agent:customer-service',
    type: 'agent',
    name: '客服助手',
    description: '处理用户咨询、投诉、退换货等客服场景',
    inputSchema: {
      type: 'object',
      properties: {
        message: { type: 'string' },
        userId: { type: 'string' },
      },
      required: ['message'],
    },
    outputSchema: {
      type: 'object',
      properties: {
        content: { type: 'string' },
        structured: { type: 'object' },
      },
    },
    tags: ['customer-service'],
  };
}

framework.register('service', CustomerServiceAgent);
```

```typescript
// 方式二：register 时传入 capability 补充信息
framework.register('service', CustomerServiceAgent, {
  description: '处理用户咨询、投诉、退换货等客服场景',
  tags: ['customer-service'],
});
```

如果 Agent 类没有 `static capability`，且 `register()` 也没有传入，则框架从 `IAgent.capabilities` 推导：

```typescript
// 自动推导示例
{
  id: 'agent:customer-service',
  type: 'agent',
  name: 'customer-service',
  description: 'Agent customer-service',
  inputSchema: AgentCapability[0].inputSchema,
  outputSchema: AgentCapability[0].outputSchema,
}
```

### 5.11.3 Tool 自动发现

Tool 注册到 Agent 时，同时注册到 `CapabilityRegistry`：

```typescript
class RefundPlugin implements IPlugin {
  install(agent: IAgent, context: PluginContext) {
    context.registerTool({
      name: 'query-order',
      description: '根据订单号查询订单状态',
      parameters: {
        type: 'object',
        properties: { orderId: { type: 'string' } },
        required: ['orderId'],
      },
    });
  }
}
```

`registerTool` 内部会调用 `CapabilityRegistry.register()`：

```typescript
{
  id: 'tool:query-order',
  type: 'tool',
  name: 'query-order',
  description: '根据订单号查询订单状态',
  inputSchema: { ... },
}
```

### 5.11.4 Skill 显式注册

Skill 不是自动发现的，必须显式注册：

```typescript
framework.discovery.register({
  id: 'skill:handle-refund',
  type: 'skill',
  name: '处理退款',
  description: '完整的退款处理流程：查询订单 → 验证条件 → 创建退款 → 通知用户',
  dependencies: ['tool:query-order', 'tool:create-refund', 'tool:send-notification'],
  tags: ['refund', 'customer-service'],
});
```

### 5.11.5 Plugin 批量注册

Plugin 可以实现 `capabilities()` 方法，一次性注册多个能力：

```typescript
class RefundPlugin implements IPlugin {
  capabilities(): Capability[] {
    return [
      { id: 'tool:query-order', type: 'tool', ... },
      { id: 'tool:create-refund', type: 'tool', ... },
      { id: 'skill:handle-refund', type: 'skill', ... },
    ];
  }
}
```

### 5.11.6 延迟发现

`framework.init()` 时，可以扫描 `agentDir` 目录，自动发现并注册所有 Agent 包：

```typescript
const framework = new AgentFramework({
  agentDir: './agents',
});

await framework.init(); // 扫描 ./agents/*/，import 每个 Agent 包并注册
```

### 5.11.7 冲突解决

如果两个能力注册时 ID 冲突：

1. **同 ID 同版本**：后注册者覆盖（或忽略，取决于策略）
2. **同 ID 不同版本**：保留版本号更高的；版本相同时后注册者覆盖
3. **冲突回调**：可配置 `onConflict: 'overwrite' | 'ignore' | 'throw'`

```typescript
interface CapabilityRegistryOptions {
  onConflict?: 'overwrite' | 'ignore' | 'throw';
}
```

`CapabilityRegistry.toPrompt()` 输出格式：


## 5.13 PlanExecutor DAG 调度

### 5.13.1 构建 DAG

PlanExecutor 根据 `PlanStep.dependsOn` 构建有向无环图（DAG）。

```typescript
interface DAGNode {
  step: PlanStep;
  inDegree: number;
  dependents: string[];
}

type DAG = Map<string, DAGNode>;
```

构建算法：

```typescript
function buildDAG(steps: PlanStep[]): DAG {
  const dag = new Map<string, DAGNode>();

  // 初始化节点
  for (const step of steps) {
    dag.set(step.id, { step, inDegree: 0, dependents: [] });
  }

  // 建立依赖关系
  for (const step of steps) {
    for (const depId of step.dependsOn || []) {
      const depNode = dag.get(depId);
      if (!depNode) throw new Error(`依赖的步骤不存在: ${depId}`);

      depNode.dependents.push(step.id);
      dag.get(step.id)!.inDegree++;
    }
  }

  // 检测环
  if (hasCycle(dag)) {
    throw new Error('ExecutionPlan 存在循环依赖');
  }

  return dag;
}
```

### 5.13.2 拓扑排序与并行调度

使用 Kahn 算法：

```typescript
async function executeDAG(dag: DAG, context: ExecutionContext): Promise<StepResult[]> {
  const results = new Map<string, StepResult>();
  const queue = Array.from(dag.values()).filter(n => n.inDegree === 0);

  while (queue.length > 0) {
    // 取出所有当前可执行的步骤（无依赖或依赖已完成）
    const readyNodes = queue.splice(0);

    // 并行执行
    const stepResults = await Promise.all(
      readyNodes.map(node => executeStep(node.step, context, results))
    );

    for (const result of stepResults) {
      results.set(result.stepId, result);

      if (!result.success) {
        // 失败处理：重试或重新规划
        await handleStepFailure(result, dag, context, results);
      }

      // 减少下游步骤入度
      const node = dag.get(result.stepId)!;
      for (const dependentId of node.dependents) {
        const dependent = dag.get(dependentId)!;
        dependent.inDegree--;
        if (dependent.inDegree === 0) {
          queue.push(dependent);
        }
      }
    }
  }

  return Array.from(results.values());
}
```

### 5.13.3 变量绑定

执行步骤前，PlanExecutor 解析 `input` 中的变量引用：

```typescript
function resolveInput(
  input: Record<string, any>,
  results: Map<string, StepResult>,
  userInput: Record<string, any>,
): Record<string, any> {
  return deepMapValues(input, (value) => {
    if (typeof value !== 'string') return value;

    // ${stepId.output.field}
    const match = value.match(/^\$\{([^.]+)\.output(?:\.(.+))?\}$/);
    if (match) {
      const [, stepId, path] = match;
      const result = results.get(stepId);
      if (!result) throw new Error(`引用的步骤结果不存在: ${stepId}`);
      return path ? get(result.output, path) : result.output;
    }

    // ${userInput.field}
    const userMatch = value.match(/^\$\{userInput(?:\.(.+))?\}$/);
    if (userMatch) {
      const [, path] = userMatch;
      return path ? get(userInput, path) : userInput;
    }

    return value;
  });
}
```

### 5.13.4 输出捕获

每个步骤执行后，结果以 `outputAs` 指定的名称保存：

```typescript
const stepResult: StepResult = {
  stepId: step.id,
  success: result.success,
  output: result.output,
  duration: Date.now() - startTime,
};

results.set(step.outputAs || step.id, stepResult);
```

下游可以通过 `${outputAs.output.field}` 或 `${stepId.output.field}` 引用。

### 5.13.5 超时处理

```typescript
interface StepExecutionOptions {
  stepTimeout?: number;       // 单步超时，默认 30000ms
  planTimeout?: number;       // 整个计划超时
}
```

单步超时触发重试；全局超时触发整个计划失败。

### 5.13.6 失败分类

| 错误类型 | 示例 | 处理方式 |
|---|---|---|
| 可重试 | Provider 429 / 网络超时 | PlanExecutor 内部重试 |
| 可重新规划 | 业务规则不满足 | 调用 PlannerAgent.replan |
| 不可恢复 | 能力不存在 / 输入校验失败 | 直接返回失败 |

## 5.14 重新规划策略

### 5.14.1 触发条件

重新规划在以下情况触发：

- 步骤返回 `success: false`
- 步骤超时
- 步骤输出不符合 `outputSchema`
- 依赖的步骤结果缺失

### 5.14.2 重新规划上下文

```typescript
interface ReplanContext {
  task: AgentTask;                    // 原始任务
  plan: ExecutionPlan;                // 当前计划
  failedStep: StepResult;             // 失败的步骤
  completedSteps: StepResult[];       // 已成功的步骤
  remainingSteps: PlanStep[];         // 未执行的步骤
  attemptNumber: number;              // 第几次重新规划
}
```

### 5.14.3 重新规划次数限制

```typescript
interface OrchestrateOptions {
  maxReplanAttempts?: number;  // 默认 3
}
```

超过最大次数后，PlanExecutor 返回失败结果。

### 5.14.4 保留成功结果

重新规划时，PlannerAgent 会收到 `completedSteps`，新计划应该：

1. 尽量不重复已成功步骤
2. 如果需要替换某步，用新的 capability 或参数
3. 新步骤的 `input` 可以引用 `completedSteps` 的输出

### 5.14.5 避免无限循环

PlanExecutor 维护 `replanHistory`：

```typescript
interface ReplanHistory {
  failedStepId: string;
  errorCode: string;
  attemptNumber: number;
  timestamp: number;
}
```

如果同一个步骤因为相同错误连续失败超过 2 次，直接终止并返回失败。

## 5.15 人工确认机制

### 5.15.1 触发条件

当 `ExecutionPlan.constraints.requireApproval === true` 时，PlanExecutor 在执行前暂停，等待确认。

`requireApproval` 自动为 true 的情况：

1. 计划包含标记为高风险的能力
2. 用户显式设置 `requireApproval: true`
3. 计划步骤数超过阈值（可配置）

### 5.15.2 高风险能力标记

```typescript
interface Capability {
  // ... 其他字段
  riskLevel?: 'low' | 'medium' | 'high';
  sensitiveOperations?: string[]; // 如 ['refund', 'contract-sign', 'payment']
}
```

高风险能力示例：

```typescript
framework.discovery.register({
  id: 'tool:create-refund',
  type: 'tool',
  name: '创建退款',
  riskLevel: 'high',
  sensitiveOperations: ['refund'],
});
```

### 5.15.3 确认回调接口

```typescript
interface ApprovalHandler {
  (plan: ExecutionPlan): Promise<ApprovalResult>;
}

interface ApprovalResult {
  approved: boolean;
  modifiedPlan?: ExecutionPlan;  // 用户修改后的计划
  reason?: string;               // 拒绝原因
}
```

使用示例：

```typescript
const framework = new AgentFramework({
  approvalHandler: async (plan) => {
    // 发送给 Dashboard 或消息队列等待人工确认
    const approved = await waitForHumanApproval(plan);
    return { approved };
  },
});

const result = await framework.orchestrate(task, { requireApproval: true });
```

### 5.15.4 Dashboard 确认 UI

Dashboard 调试台在 `/playground` 页面展示计划确认对话框：

- 显示计划目标、步骤列表、调用的能力
- 高亮高风险步骤
- 允许用户编辑步骤参数
- 提供"确认执行" / "拒绝" / "修改后执行" 按钮
