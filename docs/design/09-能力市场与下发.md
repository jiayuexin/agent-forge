## 9.1 能力分层模型

Capability Hub 管理的能力按类型分为五类：

```text
┌─────────────────────────────────────────┐
│  Layer 5: Agent（可编排 Agent）          │
│  · 进程内 StatelessAgent                 │
│  · 由编排器在 SDK 运行时直接实例化       │
├─────────────────────────────────────────┤
│  Layer 4: Remote-Agent（远端 ClientAgent）│
│  · 通过网络连接到 Capability Hub 的节点  │
│  · 可像本地能力一样被编排器远程调用      │
├─────────────────────────────────────────┤
│  Layer 3: Plugin（沙箱化脚本）           │
│  · 可执行逻辑                            │
│  · 运行在受限沙箱中                      │
│  · 需要签名 + 用户授权                   │
├─────────────────────────────────────────┤
│  Layer 2: Skill（能力组合）              │
│  · Tool 列表 + Prompt 模板               │
│  · 无代码，纯配置                        │
├─────────────────────────────────────────┤
│  Layer 1: Tool（工具定义）               │
│  · JSON Schema + 执行端点声明            │
│  · 具体执行逻辑在 Agent 本地             │
└─────────────────────────────────────────┘
```

### 各层说明

| 类型 | 定义内容 | 是否含代码 | 下发方式 | 安全级别 |
|---|---|---|---|---|
| Tool | JSON Schema + 执行端点 | 否 | JSON 定义 | 低 |
| Skill | Tool 列表 + Prompt 模板 | 否 | JSON 定义 | 低 |
| Plugin | 沙箱化脚本 | 是 | 签名脚本包 | 高 |
| Agent | 可编排的 StatelessAgent 类/工厂 | 是 | 包依赖或本地注册 | 中 |
| Remote-Agent | 已注册到 Capability Hub 的 ClientAgent 节点引用 | 否 | 节点元数据 | 中 |

## 9.2 Tool 定义

基础 `Capability` 类型定义见 [01-核心设计.md §1.14](./01-核心设计.md#114-模型驱动编排类型)。以下分别定义 Tool、Skill、Plugin 三种具体形态。

```typescript
interface ToolCapability extends Capability {
  type: 'tool';
  /** 工具执行端点类型 */
  endpointType: 'local-command' | 'local-function' | 'http' | 'remote-agent';
  /** 端点声明 */
  endpoint: {
    /** 本地函数引用或命令模板 */
    target: string;
    /** 调用方式 */
    method?: 'exec' | 'call' | 'post' | 'get';
  };
  /** 输入参数 JSON Schema */
  inputSchema: JSONSchema;
  /** 输出结果 JSON Schema */
  outputSchema?: JSONSchema;
}
```

### 示例

```json
{
  "id": "tool-git-status",
  "type": "tool",
  "name": "git-status",
  "description": "查看当前 Git 仓库状态",
  "riskLevel": "low",
  "endpointType": "local-command",
  "endpoint": {
    "target": "git status --porcelain",
    "method": "exec"
  },
  "inputSchema": {
    "type": "object",
    "properties": {
      "cwd": { "type": "string", "description": "工作目录" }
    }
  },
  "outputSchema": {
    "type": "object",
    "properties": {
      "status": { "type": "string" }
    }
  }
}
```

### ToolCapability → ToolDefinition 映射

本地 ClientAgent 收到 Hub 下发的 `ToolCapability` 后，按以下规则转换为 [01-核心设计.md §1.4](./01-核心设计.md#14-agentconfig-配置模型) 定义的 `ToolDefinition`：

| ToolCapability 字段 | ToolDefinition 字段 | 说明 |
|---|---|---|
| `name` | `name` | 直接映射，保持 kebab-case |
| `description` | `description` | 直接映射 |
| `inputSchema` | `parameters` | JSON Schema 参数定义 |
| `endpointType` | `endpointType` | 直接映射 |
| `endpoint` | `endpoint` | 直接映射 |
| `riskLevel` | — | 不下发到 ToolDefinition，由执行前安全策略单独评估 |

若 `ToolDefinition.handler` 未提供，ClientAgent 运行时将按 `endpointType` 调度执行：
- `local-command`：通过本地 shell 执行 `endpoint.target`
- `local-function`：调用本地已注册的函数 `endpoint.target`
- `http`：发起 HTTP 请求
- `remote-agent`：通过 Capability Hub 转发到指定节点

### ToolDefinition → Capability 映射

当 Tool 通过 `PluginContext.registerTool()` 注册到 `CapabilityRegistry` 时，按以下规则转换为 [01-核心设计.md §1.14](./01-核心设计.md#114-模型驱动编排类型) 定义的 `Capability`（具体为 `ToolCapability`）：

| ToolDefinition 字段 | Capability 字段 | 说明 |
|---|---|---|
| `name` | `id` | `tool-${name}`，与 [08-客户端Agent与无状态Agent.md §8.8](../design/08-客户端Agent与无状态Agent.md#88-命名约定) 的能力 ID 命名约定一致 |
| `name` | `name` | 直接复用作为展示名 |
| `description` | `description` | 直接映射 |
| `parameters` | `inputSchema` | JSON Schema 参数定义 |
| `endpointType` / `endpoint` | `endpointType` / `endpoint` | 直接映射 |
| — | `type` | 固定为 `'tool'` |
| — | `riskLevel` / `sensitiveOperations` | 由注册方显式提供，默认 `riskLevel: 'low'` |

## 9.3 Skill 定义

```typescript
interface SkillCapability extends Capability {
  type: 'skill';
  /** 该 Skill 包含的 Tool id 列表 */
  tools: string[];
  /** Skill 使用的系统 Prompt 片段 */
  promptTemplate: string;
  /** Skill 执行示例 */
  examples?: Array<{ input: unknown; output: unknown }>;
}
```

### 示例

```json
{
  "id": "skill-git-commit",
  "type": "skill",
  "name": "git-commit",
  "description": "根据代码改动生成 commit message 并提交",
  "riskLevel": "medium",
  "tools": ["tool-git-status", "tool-git-diff", "tool-git-commit"],
  "promptTemplate": "你是一个 Git 专家。请根据用户提供的改动描述生成符合 Conventional Commits 规范的 commit message...",
  "sensitiveOperations": ["git-commit"]
}
```

## 9.4 Plugin 定义

```typescript
interface PluginCapability extends Capability {
  type: 'plugin';
  /** 插件包下载地址 */
  downloadUrl: string;
  /** 包签名 */
  signature: string;
  /** 公钥指纹，用于定位对应公钥 */
  keyId: string;
  /** 插件入口文件 */
  entry: string;
  /** 插件可访问的能力白名单 */
  allowedCapabilities?: string[];
  /** 插件运行沙箱配置；类型定义见 [10-安全模型.md §10.3](./10-安全模型.md#103-能力下发安全) */
  sandbox?: SandboxConfig;
}
```

### 安全要求

- Plugin 必须经 Capability Hub 管理员签名。
- Plugin 运行在受限沙箱中，默认无网络访问。
- Plugin 只能访问声明的 `allowedCapabilities` 白名单。
- 敏感 Plugin 安装前需 ClientAgent 本地确认。

## 9.5 Capability Hub 中的版本管理

每个 Capability 都有版本号，遵循语义化版本：

```typescript
interface CapabilityVersion {
  capabilityId: string;
  version: string;
  /** 版本状态：draft / active / deprecated / revoked */
  status: 'draft' | 'active' | 'deprecated' | 'revoked';
  /** 版本变更说明 */
  changelog?: string;
  /** 是否兼容上一版本 */
  compatibleWith?: string[];
}
```

### 版本策略

- 同一 Capability 可同时存在多个 active 版本。
- ClientAgent 默认使用最新 active 版本。
- 管理员可强制 ClientAgent 回退到指定版本。
- revoked 版本不会被新安装，但已安装的版本可继续运行直到被替换。

## 9.6 能力下发协议

### 下发流程

```text
Capability Hub
      │ POST /capabilities/:id/distribute { nodeIds: [...] }
      ▼
WebSocket ControlMessage(type='capability-distribute')
      ▼
ClientAgent
      ├── 1. 校验签名
      ├── 2. 下载能力包（Plugin 需要）
      ├── 3. 安装到本地缓存
      ├── 4. 更新 manifest
      └── 5. 上报 capability-ack
```

### 下发载荷

`CapabilityDistributePayload` 类型定义见 [01-核心设计.md §1.15](./01-核心设计.md#115-远程控制与客户端运行时类型)。本节仅保留流程说明，不重复声明类型。

### 确认消息

`CapabilityAckPayload` 类型定义见 [01-核心设计.md §1.15](./01-核心设计.md#115-远程控制与客户端运行时类型)。

## 9.8 断网使用规则

- ClientAgent 启动时加载本地缓存 manifest。
- 执行能力时优先使用本地缓存定义。
- 如果本地不存在某能力，且当前离线，则返回 `AgentResult.error`，错误码 `CAPABILITY_NOT_CACHED`。
- 重新联网后，ClientAgent 主动向 Capability Hub 同步能力版本差异。

---

## 9.9 能力冲突解决

当 Capability Hub 下发的 Capability 与本地缓存冲突时：

| 场景 | 处理方式 |
|---|---|
| 同 ID 同版本 | 忽略 |
| 同 ID 新版本 | 覆盖本地，保留旧版本备份 |
| 同 ID 旧版本（Hub 回退） | 覆盖本地，回退到指定版本 |
| 依赖缺失 | 拒绝安装，返回错误 |
| 签名无效 | 拒绝安装，返回错误 |

## 9.10 能力市场

Capability Hub 提供能力市场界面，用于：

- 浏览团队内部可用的 Tool / Skill / Plugin。
- 查看能力文档、版本历史、风险等级。
- 一键下发能力到指定 ClientAgent 或 ClientAgent 组。
- 审批能力发布请求。

> 能力市场面向团队内部或受信任分发，不做公开社区市场。
